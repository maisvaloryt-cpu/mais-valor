[BUG-C1] PREÇO MÉDIO INCORRETO APÓS VENDA PARCIAL
Arquivo : common.js — função calcPosicoes(), linhas 301-303
Código problemático:
  if(isSell){
    map[t].qtd = Math.max(0, map[t].qtd - a.qtd);
    map[t]._custo = map[t].qtd * map[t].pm;   <- problema aqui
  }

Problema:
  A linha recalcula _custo DEPOIS de já ter subtraído a qtd. Isso parece
  correto na aparência (custo restante = novas_cotas * PM), mas esconde um
  erro sutil: se houver uma venda parcial SEGUIDA de nova compra, o _custo
  acumulado pode divergir do real. A forma mais robusta seria:

    // antes de subtrair a qtd:
    const qtdRestante = Math.max(0, map[t].qtd - a.qtd);
    map[t]._custo = qtdRestante * map[t].pm;
    map[t].qtd = qtdRestante;

  A sequência atual faz a subtração e depois usa o .qtd já atualizado para
  calcular _custo, o que produz o mesmo resultado SOMENTE se não houver
  compra nova logo depois. Na implementação atual isso funciona, mas a
  lógica é frágil e propensa a erros se alguém mexer na ordem das linhas.

Impacto: MÉDIO — pode gerar PM errado em carteiras com muitas compras/vendas.


[BUG-C2] DY (DIVIDEND YIELD) CALCULADO SOBRE CUSTO, NÃO SOBRE VALOR DE MERCADO
Arquivo : common.js — função calcAtivos(), linhas 321-323
Código:
  const proventos = calcProventosAtivo(a.ticker, a.data, a.qtd);
  const dy = custo > 0 ? (proventos / custo) * 100 : 0;

Problema:
  O DY correto (padrão de mercado) é calculado sobre o PREÇO ATUAL (valor
  de mercado), não sobre o custo de aquisição. O que o código calcula é o
  "yield on cost" (YoC), não o DY.

  Fórmula correta: dy = vt > 0 ? (proventos / vt) * 100 : 0;
  Fórmula implementada: dy = custo > 0 ? (proventos / custo) * 100 : 0;

  Isso faz com que ativos que valorizaram muito (vt >> custo) pareçam ter
  um DY irreal — por exemplo, um ativo comprado a R$10 que hoje vale R$50
  e pagou R$2 de dividendos/acao apareceria com DY de 20% (sobre custo)
  quando o DY real é 4% (sobre valor atual).

Impacto: ALTO — distorce completamente a visão de renda dos ativos.


[BUG-C3] DY MÉDIO DA CARTEIRA CALCULADO SEM PONDERAÇÃO CORRETA
Arquivo : common.js — função getTotals(), linha 348
Código:
  const provAno = calcProventosUltimos12m();
  const dyMed = total > 0 ? (provAno / total) * 100 : 0;

Problema:
  Embora a fórmula seja financeiramente razoável (renda anual / patrimônio
  atual = DY anualizado), o campo "DY médio" exibido no resumo.html é uma
  MÉDIA PONDERADA INCORRETA. Ele simplesmente divide os proventos pelo
  total da carteira, o que ignora que alguns ativos não pagam proventos.
  O correto seria o DY ponderado pelo peso de cada ativo na carteira:

    dyMed = Soma( dy_i * peso_i ) para ativos com dy > 0

  Adicionalmente, calcProventosUltimos12m() usa a data atual menos 1 ano
  como cutoff. Para ativos com poucos meses de carteira, o numerador inclui
  apenas proventos parciais, mas o denominador usa o patrimônio total atual,
  o que subdeclara o DY dos ativos novos.

Impacto: MÉDIO — distorce o DY médio apresentado nos cards de métricas.


[BUG-C4] RENTABILIDADE TOTAL SEM ANUALIZAÇÃO (SEM CAGR)
Arquivo : common.js — função calcAtivos(), linha 324
Código:
  const rentTotal = custo > 0 ? (lucroTotal / custo) * 100 : 0;

Problema:
  A rentabilidade exibida como "Rentabilidade Total" é a rentabilidade
  absoluta acumulada desde a compra, sem nenhuma anualização. Um ativo
  comprado há 5 anos com +30% de retorno aparece com "30%" assim como um
  ativo comprado há 1 mês que também rendeu 30%.

  Sem anualização (CAGR ou taxa mensal), a métrica é enganosa para carteiras
  com ativos de diferentes idades. Deveria existir ao menos uma coluna de
  "Rent. Anualizada" ou deixar claro no label que é rentabilidade acumulada.

Impacto: MÉDIO — métrica apresentada sem contexto temporal, pode induzir
          decisões erradas de alocação.


[BUG-C5] GANHO REALIZADO: PM PODE ESTAR INCORRETO POR CONTA DO FILTRO DE DATA
Arquivo : common.js — função getGanhosRealizados(), linhas 255-261
Código:
  vendas.reduce((total, venda) => {
    const txAntes = ativos.filter(a => a.ticker === venda.ticker
                                    && a.data && a.data < venda.data);
    const pos = calcPosicoes(txAntes);
    const pmCompra = pos.find(p => p.ticker === venda.ticker)?.pm || 0;
    return total + (venda.pm - pmCompra) * venda.qtd;
  }, 0);

Problema 1 — Filtro por data ignora horário (apenas YYYY-MM-DD):
  Se houver duas vendas no mesmo dia para o mesmo ticker, o filtro
  'a.data < venda.data' pode incluir ou excluir incorretamente, pois a
  comparação é por string de data (sem hora).

Problema 2 — Recalcula calcPosicoes() para CADA venda individualmente:
  Isso é O(N²) mas principalmente pode incluir vendas anteriores DO MESMO
  ANO no cálculo, levando a um PM incorreto para vendas subsequentes.
  Exemplo: 3 compras + 2 vendas no mesmo dia → o PM da 2ª venda pode ser
  calculado com a 1ª venda já incluída nas "txAntes", puxando o PM errado.

Problema 3 — Custo para IRPF deve ser o custo MÉDIO conforme IN 1.585:
  A norma da Receita Federal exige o método do custo médio ponderado.
  O código tenta isso, mas a reconstituição da posição por calcPosicoes
  pode falhar se lançamentos não tiverem data preenchida (data='').

Impacto: ALTO — erro direto no cálculo de IRPF. Pode levar ao pagamento
          incorreto de DARF.


[BUG-C6] PROVENTOS CALCULADOS PELA DATA DE PAGAMENTO, NÃO PELA DATA EX-DIVIDENDO
Arquivo : common.js — função calcProventosAtivo(), linhas 176-188
Código:
  return cache
    .filter(d => d.date >= from)
    .reduce((s, d) => s + d.value * getQtdTickerAtDate(ticker, d.date), 0);

Problema:
  getQtdTickerAtDate() verifica a posição na data do dividendo (d.date).
  Se d.date é a data de PAGAMENTO (e não a data EX-DIVIDENDO), o sistema
  pode creditar proventos para ações compradas DEPOIS da data ex, antes
  do pagamento. Isso superestima os proventos recebidos.

  A data relevante para apurar direito ao provento é a data COM (ex-data),
  não a data de pagamento. Como o campo 'date' nos dados pode ser a data de
  pagamento, o sistema pode estar creditando proventos indevidos.

Impacto: ALTO — superestima proventos. Depende da qualidade do dado fonte.


[BUG-C7] CUSTO DE AQUISIÇÃO NO IRPF USA POSIÇÃO ATUAL, NÃO 31/12 DO ANO-CALENDÁRIO
Arquivo : irpf.html — função renderIRPF(), linhas 70-82
Código:
  const c = calcAtivos();   <- usa a posição ATUAL, não de 31/12/ano-calendário
  ...
  document.getElementById('tbody-irpf').innerHTML = c.map(a => `
    <td>${fmtR(a.custo)}</td>   <- custo ATUAL (inclui compras do ano corrente)
  `).join('');

Problema:
  Para o IRPF, a declaração de "Bens e Direitos" deve refletir a posição em
  31/12 do ano-calendário, com o custo de aquisição daquele momento.
  O sistema mostra a posição ATUAL (hoje), que pode incluir compras feitas
  em 2025/2026 que não devem constar na declaração de 2024.

  A função calcAtivos() não aceita parâmetro de data de corte.
  O correto seria filtrar: ativos onde a.data <= '2024-12-31'.

Impacto: CRÍTICO — dado errado para declaração de IRPF.


[BUG-C8] EVOLUÇÃO DO PATRIMÔNIO: CUSTO ZERADO QUANDO SEM PREÇO HISTÓRICO
Arquivo : common.js — função calcEvolucaoPatrimonio(), linhas 131-138
Código:
  posicoes.forEach(a => {
    ...
    if(price){ vt += a.qtd * price; custo += a.qtd * a.pm; }
    // se nao tiver price, custo nao é somado!
  });

Problema:
  O custo só é somado se houver preço histórico disponível (`price`).
  Se um ativo não tem preço histórico naquele mês (ex.: Renda Fixa,
  ativo recente ou ativo sem dados no site), a sua qtd é ignorada no custo.
  O gráfico de evolução mostrará custo menor do que o real.

  Além disso, o filtro `!a.data || a.data.slice(0,7) <= ym` inclui
  lançamentos SEM DATA em todos os meses, distorcendo o histórico.

Impacto: MÉDIO — gráfico de evolução do patrimônio subestima custo real.

[BUG-L2] DEDUPLICAÇÃO DE IMPORTAÇÃO PODE FALHAR COM FLOATS
Arquivo : common.js — função processImportRows(), linhas 427-428
Código:
  const key = `${obj.ticker}|${obj.tipo}|${obj.data}|${obj.qtd}|${obj.pm}`;
  if(ativos.some(a => `${a.ticker}|${a.tipo}|${a.data}|${a.qtd}|${a.pm}` === key)) continue;

Problema:
  A chave de deduplicação usa floats como string. Dois números podem ser
  matematicamente iguais mas produzir strings diferentes por erro de
  arredondamento de ponto flutuante:
    28.1 + 0.2 → "28.299999999999997" vs "28.5" (falha na comparação)

  Se o preço vier do Excel com arredondamento, a deduplicação não funciona
  e o mesmo lançamento pode ser importado duplicado.

  Solução: const key = `${ticker}|${tipo}|${data}|${qtd.toFixed(4)}|${pm.toFixed(4)}`;

Impacto: MÉDIO — duplicação de lançamentos ao reimportar Excel.


[BUG-L3] IMPORTAÇÃO B3: DETECÇÃO DA COLUNA "ENTRADA/SAÍDA" É FRÁGIL
Arquivo : common.js — função processB3Rows(), linhas 480-482
Código:
  if(k.includes('entrada') || k.includes('sa')) col.tipo = idx;

Problema:
  A condição `k.includes('sa')` vai capturar QUALQUER coluna cujo nome
  contenha "sa" (ex.: "data", "saldo", "casa"). Isso pode mapear a coluna
  errada para o tipo de operação (compra/venda), invertendo a classificação.

  O correto seria verificar o nome exato esperado do extrato B3:
    if(k === 'entrada/saída' || k.startsWith('entrada') || k === 'tipo')

Impacto: MÉDIO — importação do extrato B3 pode classificar compras como
          vendas ou vice-versa.


[BUG-L4] VENDA A DESCOBERTO IGNORADA SILENCIOSAMENTE
Arquivo : common.js — função calcPosicoes(), linha 302
Código:
  map[t].qtd = Math.max(0, map[t].qtd - a.qtd);

Problema:
  O sistema previne que a qtd vá negativa, mas silenciosamente ignora o
  excesso vendido. Se o usuário vender mais do que tem (erro de digitação),
  o sistema corta em 0 sem aviso — o lançamento de venda fica registrado
  normalmente, gerando dados inconsistentes (venda aparece nos lançamentos
  mas não afeta a posição).

Impacto: BAIXO-MÉDIO — inconsistência silenciosa nos dados.

[BUG-P1] ACUMULAÇÃO DE PROVENTOS SEM ARREDONDAMENTO INTERMEDIÁRIO
Arquivo : common.js — funções calcProventosAtivo() e getDividendosPorMes()
Código:
  .reduce((s, d) => s + d.value * getQtdTickerAtDate(ticker, d.date), 0);

Problema:
  Multiplicações repetidas de float (d.value * qtd) acumulam erros de
  ponto flutuante. Em carteiras grandes (muitos tickers, muitos dividendos),
  a soma pode ter centavos de erro (ex.: R$ 1.000,0000000000001).
  Em relatórios fiscais, isso pode gerar inconsistências.

  Solução: arredondar cada parcela:
    .reduce((s, d) => s + Math.round(d.value * qtd * 100) / 100, 0)

Impacto: BAIXO — erro de centavos, visível em relatórios fiscais.


[BUG-P2] fmt() MASCARA NaN COMO ZERO
Arquivo : common.js — linha 57
Código:
  const fmt = (v, d=2) => (v || 0).toLocaleString('pt-BR', {...});

Problema:
  O operador `||` trata NaN como falsy, retornando 0 quando deveria
  indicar ausência de dado. Isso mascara erros de cálculo que produzem
  NaN (ex.: 0/0 antes da guarda de custo>0).

  Melhor: const fmt = (v, d=2) => (isNaN(v)||v==null ? 0 : v).toLocaleString(...)

Impacto: BAIXO — mascara NaN como zero, pode esconder bugs de cálculo.


[BUG-P3] GRÁFICO DE EVOLUÇÃO SEM MENSAGEM "SEM DADOS"

[BUG-F1] CÓDIGO IRPF DE BDR EXIBIDO INCORRETAMENTE NA TELA
Arquivo : common.js linha 6, irpf.html linha 82
Código:
  const IRPF_CODE = { ..., 'BDR': '04', ... };
  // irpf.html exibe:
  `Grupo ${IRPF_CODE[a.classe] || '03'}`

[BUG-F2] TRIBUTAÇÃO DE ETFs E EXTERIOR INCOMPLETA
Arquivo : irpf.html — linhas 86-89
Código:
  if(a.classe === 'FII') trib = 'Isento';
  else if(a.classe === 'B3') trib = 'Div: Isento / JCP: 15%';
  else trib = 'Tributável';   <- ETFs e Exterior caem aqui

Problema:
  - ETFs de renda variável no Brasil: 15% de IR sobre lucro na venda

    rendimentos pagos por fundos no exterior podem ser tributados na fonte

  Classificar tudo como "Tributável" sem alíquota é insuficiente e pode
  induzir erro no cálculo de IR a recolher.

Impacto: ALTO — pode induzir o usuário a calcular DARF incorretamente.


[BUG-F3] ISENÇÃO MENSAL DE CRIPTO NÃO É CALCULADA
Arquivo : common.js — função getGanhosRealizados()
Problema:
  Criptomoedas têm isenção de IR se vendas mensais ficarem abaixo de
  R$35.000 (mesmo limite de ações listadas em bolsa para pessoa física).
  getGanhosRealizados() soma os ganhos do ano todo sem separar por mês

  cripto pode ser menor do que o valor exibido.

Impacto: MÉDIO — usuário pode calcular DARF de cripto a maior.


[BUG-F4] EXPORTAÇÃO IRPF USA PROVENTOS ESTIMADOS EM VEZ DE REAIS
Arquivo : common.js — função exportarRelatorioIRPF(), linhas 380-382
Código:
  rendimentosIsentos: c.map(a => ({
    dyAnual: a.dy,
    proventoEstAnual: a.vt * (a.dy / 100),   <- ESTIMATIVA, não valor real
  }))

Problema:
  O campo exportado `proventoEstAnual` é uma estimativa: valor de mercado
  atual * DY atual. O IRPF exige os proventos EFETIVAMENTE recebidos,
  disponíveis em `a.proventos` (calculado por calcProventosAtivo).

  Além disso, como DY é calculado sobre custo (BUG-C2), a estimativa de
  provento anual fica duplamente errada.

  Correção: usar a.proventos no lugar de a.vt * (a.dy / 100).

Impacto: CRÍTICO — relatório IRPF exportado contém valores estimados
          onde deveria ter valores reais recebidos.

[BUG-U1] TABELA DE RESUMO: COLUNA "RENTABILIDADE" EXIBE SÓ VARIAÇÃO DE CAPITAL
Arquivo : resumo.html — linhas 199-200
Código:
  <td class="r"><span class="pill...">${fmt(Math.abs(a.resPct))}%</span></td>
  <td class="r ...>${fmtP(a.resPct)}</td>   <- a.resPct (só capital), nao a.rentTotal

Problema:
  A coluna tem label "Rentabilidade" mas exibe `resPct` (variação de capital
  sem proventos). A rentabilidade real incluiria proventos (rentTotal).
  O usuário vê uma rentabilidade que ignora dividendos recebidos.

Impacto: MÉDIO — usuário interpreta rentabilidade como total quando é só capital.


[BUG-U2] METAS SEM VALIDAÇÃO DE SOMA IGUAL A 100%
Arquivo : metas.html, common.js — função addMeta()
Problema:
  O sistema permite salvar metas cuja soma ultrapasse 100% ou fique abaixo
  de 100% sem nenhum aviso. A barra de progresso exibe valores incoerentes.
  Exemplo: adicionar "BDR: 15%" ao DEFAULT (que já soma 100%) resulta em
  115% total e diferenças positivas irreais para todas as classes.

Impacto: BAIXO — aviso de UX ausente, não corrompe dados mas confunde.


[BUG-U3] "CORRELAÇÃO POR CLASSE" É NA VERDADE ALOCAÇÃO POR CLASSE
Arquivo : analise.html — linha 35 e linha 80
Problema:
  O gráfico polar é chamado de "Correlação por Classe" mas exibe o
  percentual de alocação (peso) de cada classe na carteira. Correlação é
  uma métrica estatística diferente (coeficiente de Pearson entre retornos).
  O label incorreto pode confundir investidores com conhecimento financeiro.

Impacto: BAIXO — problema de nomenclatura, nao de cálculo.


[BUG-U4] DATAS EXIBIDAS COM POSSÍVEL OFFSET DE FUSO HORÁRIO
Arquivo : lancamentos.html — linha 98
Código:
  new Date(a.data).toLocaleDateString('pt-BR')

Problema:
  `new Date('2024-03-15')` interpreta a string ISO como UTC meia-noite.
  Em fuso UTC-3 (Brasília), isso renderiza '14/03/2024' (um dia antes).
  Afeta todas as datas de lançamentos exibidas na tabela.

  Solução: parsear manualmente:
    const [y,m,d] = a.data.split('-');
    new Date(+y, m-1, +d).toLocaleDateString('pt-BR')

Impacto: MÉDIO — datas exibidas com 1 dia a menos, confunde o usuário.

[BUG-D1] ATIVOS EM USD SEM CONVERSÃO CAMBIAL — PATRIMÔNIO TOTAL INCORRETO
Arquivo : common.js — função calcAtivos(), linhas 317-319
Código:
  const cotacao = COTACOES[a.ticker] || a.cotacao || 0;
  const vt = a.qtd * cotacao;
  const custo = a.qtd * a.pm;
  // campo a.moeda existe mas nunca é usado nos cálculos!

Problema:
  Para ativos com `a.moeda = 'USD'` (Exterior, REITs), a cotação pode estar
  em dólares enquanto o PM pode estar em reais ou dólares conforme o usuário
  inseriu. Não há nenhuma conversão cambial em nenhum ponto do código.
  O patrimônio total (soma de vt de todos os ativos) mistura BRL e USD
  sem conversão, produzindo um total financeiramente incorreto.

  O campo `moeda` existe nos dados mas é decorativo — não afeta nenhum cálculo.

Impacto: CRÍTICO — patrimônio total incorreto para carteiras com ativos
          internacionais em USD.


[BUG-D2] SYNC FIREBASE SEM DEBOUNCE/BATCHING EM IMPORTAÇÕES
Arquivo : common.js — linha 41
Código:
  function saveAtivos(){
    localStorage.setItem(STORAGE_ATIVOS, JSON.stringify(ativos));
    if(typeof syncFirestore === 'function') syncFirestore();
  }

Problema:
  Toda operação de salvar dispara syncFirestore(). Em importações em lote
  (ex.: 200 lançamentos), se syncFirestore() não tiver debounce/batching
  interno, podem ocorrer múltiplas escritas simultâneas no Firestore,
  causando throttling e possível perda de dados em race conditions.

Impacto: MÉDIO — possível perda de dados em sincronização Firebase.


[BUG-D3] LANÇAMENTOS SEM DATA DISTORCEM GRÁFICO HISTÓRICO
Arquivo : common.js — função calcEvolucaoPatrimonio(), linha 128
Código:
  const txAte = ativos.filter(a => !a.data || a.data.slice(0,7) <= ym);

Problema:
  `!a.data` é true para lançamentos sem data (data='' ou undefined).
  Isso inclui esses lançamentos em TODOS os meses do histórico, como se
  fossem compras desde o início dos tempos. O gráfico de evolução mostrará
  custo distorcido retroativamente em todos os meses.

  A importação CSV pode resultar em ativos sem data se o campo vier vazio.

Impacto: MÉDIO — distorce gráfico de evolução patrimonial histórica.
